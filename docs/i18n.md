# Internationalization (English ↔ Hebrew)

The template ships a full bilingual UI. **The existing LTR/RTL toggle is the
language switch**: LTR renders English, RTL renders Hebrew. There is no separate
language picker — direction and language move together.

## How it works

- **One switch point.** `src/client/context/direction-provider.tsx` watches the
  `dir` cookie. Its effect sets `<html dir>` + `<html lang>` and calls
  `i18n.changeLanguage(lang)`, where `lang = langFromDir(dir)` (`rtl → he`,
  otherwise `en`). Flip the toggle in the config drawer and the whole UI
  re-renders in the other language; the choice survives reload via the cookie.
- **Both locales are bundled.** `src/client/i18n/index.ts` initializes
  `i18next` + `react-i18next` with `en.json` and `he.json` imported statically
  (only two languages — no HTTP backend, no lazy namespaces). The `dir` cookie
  is the detector, so there is no `i18next-browser-languagedetector`.
- **Type-safe keys.** `src/client/i18n/i18next.d.ts` augments i18next with
  `resources: { translation: typeof en }`, so `t('...')` only accepts keys that
  exist in `en.json`. `TranslationKey` (exported from `@/i18n`) is that key union
  — use it to type data arrays that hold keys.
- **Clerk chrome.** `main.tsx` passes `@clerk/localizations`' `heIL` to
  `ClerkProvider` when the `dir` cookie is `rtl` on page load, so sign-in /
  sign-up / user-button render in Hebrew. Mid-session toggles update Clerk-owned
  UI on the next load (documented ceiling).

## The hard rule: both locales move together

**Every new UI string goes in BOTH `en.json` and `he.json`.** The parity test at
`test/setup/i18n.test.mjs` fails the build if the two files have different key
sets, or if any key's `{{token}}` interpolation placeholders differ between them.
Run `pnpm test` and the parity check runs with it.

## Conventions

- **No `t()` at module scope.** Module-level arrays/consts (enum labels,
  sidebar/nav data, radio option lists) hold translation **KEYS** as strings,
  typed `TranslationKey`. Call `t(key)` at **render** in the owning component —
  calling `t()` once at module load would freeze the initial language. For
  column definitions and Zod schemas, thread `t` in via a factory
  (`getXColumns(t)` / `makeSchema(t)`) and memoize with `useMemo(..., [t])`.
- **No `__…__` tokens in locale JSON.** `'__APP_NAME__'` (the product-name
  sentinel that `setup.mjs` replaces) and `plan: 'Template'` stay literal in
  `sidebar-data.ts`. The CI sentinel scan + `test/setup/ci-gates.test.mjs` fail
  if a `__…__` token ever lands in a locale file.

## Out of scope (English / LTR by design)

- **Number & date formatting** is not localized — dates render via
  `toLocaleDateString()` and demo numbers/currency stay as literals (DV-7).
- **Chart internals** (Recharts axis ticks, legend/tooltip series labels) stay
  English and keep their `<div dir='ltr'>` wrappers so the plot canvas does not
  mirror. Only the surrounding card titles/descriptions are translated.

## Adding a string

1. Add the key to `en.json` **and** the Hebrew to `he.json` (same key path).
2. In the component: `const { t } = useTranslation()` → `{t('your.key')}`.
   Interpolate with `t('your.key', { count })` for `{{count}}`-style tokens.
3. `he.json` is flagged for native-Hebrew-speaker review — the shipped Hebrew is
   idiomatic-best-effort, not professionally reviewed.
