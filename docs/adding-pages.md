# Adding Pages, Routes & API Endpoints

The most common task in a forked dashboard is "add a page." This guide is the
copy-paste recipe book for it. The canonical worked example throughout is the
shipped **`items`** feature — every path below is real and in the repo.

Two invariants govern everything here:

1. **Protected = both layers.** A page is only truly protected when its route
   lives under `src/client/routes/_authenticated/` **and** its API router is
   mounted **after** `requireAuth` in `src/server/index.ts`. The client guard is
   UX; the server registration order is the real security boundary.
2. **Never hand-edit `src/client/routeTree.gen.ts`** (auto-generated by the
   build) and **never run `drizzle-kit push`** (this template is generate-only —
   see [`docs/data-layer.md`](data-layer.md)).

## 1. Page anatomy — the 4-file pattern

A protected, data-backed page is four client files plus its server leg. The
`items` feature is the reference:

| # | File                                                     | Role                                                        |
| - | -------------------------------------------------------- | ----------------------------------------------------------- |
| 1 | `src/client/routes/_authenticated/items/index.tsx`       | Thin route file — just wires the URL to the feature component |
| 2 | `src/client/features/items/`                             | Feature UI (`index.tsx` + `components/`)                     |
| 3 | `src/client/features/items/data/use-items.ts`            | TanStack Query hook — plain `fetch('/api/…')`               |
| 4 | `src/client/components/layout/data/sidebar-data.ts`      | Sidebar nav entry (also feeds the command palette)          |

Server side of the same example:

- `src/server/routes/items.ts` — the Hono router
- `src/server/routes/items.test.ts` — the **mandatory** colocated route test
- `src/shared/types.ts` — the shared Zod schemas + types imported by both tiers

> The thin route file is deliberately tiny:
>
> ```tsx
> import { createFileRoute } from '@tanstack/react-router'
> import { Items } from '@/features/items'
>
> export const Route = createFileRoute('/_authenticated/items/')({
>   component: Items,
> })
> ```
>
> `src/client/routeTree.gen.ts` is regenerated by the router plugin when Vite
> runs. Careful: `pnpm build` is `tsc -b && vite build` — typecheck runs
> *first*, so after adding a route file run `pnpm dev` once to regenerate the
> tree; until then `pnpm build` / `pnpm typecheck` fails on the new route path
> (`TS2345 … not assignable to keyof FileRoutesByPath`). Never edit the file by
> hand.

## 2. Recipe — a public page

A page is public when its route file lives **outside** `_authenticated/`.
Everything at `src/client/routes/` top-level (`index.tsx`, `(auth)/`,
`(errors)/`) is public.

Create `src/client/routes/pricing.tsx`:

```tsx
import { createFileRoute } from '@tanstack/react-router'

export const Route = createFileRoute('/pricing')({
  component: () => <div>Pricing</div>,
})
```

`pnpm dev` regenerates the route tree and the page is live at `/pricing` with no
sign-in required.

## 3. Recipe — a protected page (no data)

1. **Feature dir:** `src/client/features/reports/index.tsx` exporting a
   `Reports` component (compose from `src/client/components/ui/` primitives).
2. **Route file** under the guard: `src/client/routes/_authenticated/reports/index.tsx`:

   ```tsx
   import { createFileRoute } from '@tanstack/react-router'
   import { Reports } from '@/features/reports'

   export const Route = createFileRoute('/_authenticated/reports/')({
     component: Reports,
   })
   ```
3. **Sidebar entry** in `src/client/components/layout/data/sidebar-data.ts` —
   add `{ title: 'Reports', url: '/reports', icon: FileText }` to a `navGroups`
   item list (import the `lucide-react` icon at the top).

The `_authenticated/route.tsx` `beforeLoad` redirects unauthenticated visitors
to sign-in, so any route nested under `_authenticated/` inherits the guard.

## 4. Recipe — a protected page with data (full path)

This is the end-to-end path, copying the `items` patterns. Order matters.

**a. Shared contract** — add the Zod schema + type to `src/shared/types.ts` (the
one seam imported by both the server validator and the client). Follow the
`createItemSchema` / `Item` shape.

**b. Drizzle schema → migration** — edit `src/server/db/schema.ts`, then:

```bash
pnpm db:generate --name reports  # drizzle-kit generate → migrations/000N_reports.sql
pnpm db:migrate:local            # apply to the workerd/dev/test D1
pnpm db:migrate:remote           # real D1 — forked projects only (needs the real
                                 # database_id; skip on the template itself)
```

Depth on schema, seed placement, and `--local` vs `--remote` (they never sync)
lives in [`docs/data-layer.md`](data-layer.md) — don't re-derive it here.

**c. Server router** — `src/server/routes/reports.ts`, copying the `items.ts`
patterns exactly:

- **Per-request Drizzle client** — build `drizzle(c.env.__D1_BINDING__, …)`
  inside each handler. The binding only exists at request time; a module-scope
  client is a runtime error.
- **`zValidator('json', schema)`** to validate bodies (no over-posting).
- **Response envelopes** — `{ items }` / `{ item }` / `{ success: true }` /
  `{ error }`, matching the rest of the API.

```ts
export const reports = new Hono<{ Bindings: Env }>().get('/', async (c) => {
  const db = drizzle(c.env.__D1_BINDING__, { schema: { reports: reportsTable } })
  const rows = await db.select().from(reportsTable).all()
  return c.json({ reports: rows })
})
```

Pagination (`paginationQuerySchema` + `paginate` from `src/server/db/queries.ts`)
is optional and omitted above — see the `items.ts` GET for the paginated version.

**d. Mount AFTER the gate** in `src/server/index.ts`:

```ts
app.route('/api/reports', reports) // after requireAuth, before the catch-all
```

Registration order **is** the security contract (see §7). Mounting after
`requireAuth` makes the router protected by default; the terminal
`app.all('/api/*', …404)` must stay last.

**e. Mandatory colocated test** — `src/server/routes/reports.test.ts`, copying
`items.test.ts`: mount the real router behind the real `requireAuth` with
`clerkAuth` seeded to a signed-in user, and drive it with
`app.request(path, init, env)` against the real `cloudflare:test` D1. Every new
API route gets a test.

**f. Client hook** — `src/client/features/reports/data/use-reports.ts`, copying
`use-items.ts`: a plain `fetch('/api/reports')` in a `useQuery`, with a shared
`queryKey` that each mutation invalidates so the list re-fetches authoritative
server state.

**g. Route file + sidebar entry** — as in §3. Then run `pnpm dev` once so the
route tree regenerates before you run `pnpm build` / `pnpm typecheck` (see the
note in §1).

## 5. Recipe — change the post-login main page

The default landing after sign-in is `/dashboard`. It is set in
`src/client/main.tsx` on the `ClerkProvider`:

```tsx
signInFallbackRedirectUrl='/dashboard'
signUpFallbackRedirectUrl='/dashboard'
```

Change both to your new target (e.g. `/reports`). The sign-in route
(`src/client/routes/(auth)/sign-in`) carries a `forceRedirectUrl` fallback for
the "redirect back to where you were" case; update it if you change the default
landing. The `_authenticated` guard redirect target is the same seam.

## 6. Recipe — remove or rename a demo page

The base ports several **demo** feature modules from shadcn-admin. Removing one
means deleting its route file + feature dir + sidebar entry (+ any server
router). Confirmed against `sidebar-data.ts`, these demos are safe to delete:

| Demo    | Route file                                  | Feature dir                    | Server router                          |
| ------- | ------------------------------------------- | ------------------------------ | -------------------------------------- |
| `apps`  | `routes/_authenticated/apps/`               | `features/apps/`               | —                                      |
| `chats` | `routes/_authenticated/chats/`              | `features/chats/`              | —                                      |
| `tasks` | `routes/_authenticated/tasks/`              | `features/tasks/`              | —                                      |
| `users` | `routes/_authenticated/users/`              | `features/users/`              | `src/server/routes/users.ts` (+ test)  |

For each: delete the files, remove its `sidebar-data.ts` entry, and (for
`users`) unmount `app.route('/api/users', users)` in `src/server/index.ts`. Run
`pnpm dev` (regenerates the route tree) then `pnpm test`.

**Keep:** `items` (canonical CRUD example), `dashboard` (overview), `settings`
(Clerk user area), `errors`.

To **rename**, do the same delete/re-add against the new name — don't try to
patch `routeTree.gen.ts`.

## 7. Recipe — a public API endpoint

Public API routers must be mounted **before** the Clerk gate, exactly like
`/api/health`:

```ts
app.route('/api/health', health) // BEFORE clerkMiddleware() + requireAuth
```

> **⚠ Security — registration order is the real gate.** `src/server/index.ts`
> matches routers in the order they are registered. Anything mounted **before**
> `app.use('/api/*', clerkMiddleware())` + `app.use('/api/*', requireAuth)` is
> public; anything **after** is protected. A public endpoint mounted after the
> gate is unreachable; a "protected" endpoint accidentally mounted before it is
> wide open. When in doubt, read the authoritative inline comments in
> `src/server/index.ts` — they document this contract at each numbered step.

## 8. Recipe — KV-backed page data

For a single JSON blob (preferences, feature flags) rather than relational rows,
copy the **settings** pattern: `src/server/routes/settings.ts` +
`src/shared/settings.ts`. Read with `settingsSchema.safeParse` and fall back to
`defaultSettings` — the KV blob is untrusted (it may be a stale shape across
deploys), so always re-validate on read. Full detail is in the KV section of
[`docs/data-layer.md`](data-layer.md); don't duplicate it.
